# Jaeger - руководство по использованию

## Обзор

Jaeger — это система распределённой трассировки для мониторинга и устранения неполадок в микросервисных архитектурах.

### Основные функции

- **Распределённая трассировка** — отслеживание запросов через все сервисы
- **Анализ производительности** — выявление медленных операций
- **Мониторинг зависимостей** — визуализация связей между сервисами
- **Поиск ошибок** — быстрое нахождение проблем

---

## Запуск

### Предварительные требования

Jaeger запускается вместе с приложением через Docker Compose.

### Запуск приложения

```bash
# Пересобрать и запустить все сервисы
./gradlew clean build -x test
docker compose -f docker-compose.dev.yml up -d --build

# Или запустить только Jaeger с существующими сервисами
docker compose -f docker-compose.dev.yml up -d jaeger-dev
```

### Доступ к Jaeger UI

Откройте в браузере: **http://localhost:16686**

## Основные понятия

### Trace (Трассировка)

Полный путь одного HTTP-запроса от начала до конца через все сервисы.

```
trace-id: abc123
│
├── span 1:  api-gateway        50ms
├── span 2:  auth-service      30ms
├── span 3:  user-service    100ms
└── span 4:  database        80ms
```

### Span (Спан)

Отдельный этап выполнения внутри одного сервиса.

```
Timeline View:
[██████████░░░░░░░░░░░░░] api-gateway     50ms
[████████░░░░░░░░░░░░░░] auth-service  30ms
[███████████████████░░░░] database     80ms (медленно!)
```

## Поиск и анализ трассировок

### Поиск трассировок

1. Выберите **Service** из выпадающего списка
2. При необходимости выберите **Operation** (конкретны�� endpoint)
3. Укажите период в **Lookback**
4. Нажмите **Find Traces**

### Доступные сервисы

- `api-gateway` — входная точка всех запросов
- `authentication-service` — аутентификация и авторизация
- `user-management-service` — управление пользователями
- `quest-management-service` — управление квестами
- `game-engine-service` — игровой движок
- `team-management-service` — управление командами
- `notification-service` — уведомления
- `file-storage-service` — хранение файлов
- `statistics-service` — статистика

### Просмотр результатов

После поиска отображается список трассировок:

```

│  Time    │  Name            │  Duration  │  Spans  │  Errors  │

│  14:28  │  POST /login   │  89ms      │  8      │  0      │
│  14:27  │  GET /users    │  203ms     │  11     │  🔴 1    │
```

### Timeline View (Временная диаграмма)

Кликните на трассировку для детального просмотра:

```
Timeline                     Duration    Tags
│                            │          │
├── http.post                 50ms       http.url=/api/quests
│   └── auth-service          40ms       user.id=123
│       └── database         35ms       sql=SELECT *
│
├── http.server               45ms        [total]
```

### Индикация времени

- 🔵 Синий — нормальное время выполнения
- 🟡 Жёлтый — немного медленно (до 1 секунды)
- 🔴 Красный — медленно (более 1 секунды) или ошибка

## Типичные сценарии использования

### Сценарий 1: Медленный API-запрос

**Проблема:** Пользователь жалуется на медленный ответ API.

**Решение:**

1. Выберите соответствующий сервис (например, `quest-management-service`)
2. Найдите проблемный endpoint (например, `POST /quests`)
3. Отсортируйте по Duration (нажмите на заголовок)
4. Кликните на самую медленную трассировку
5. Посмотрите Timeline — какой этап занимает больше всего времени
6. Оптимизируйте самый медленный компонент

### Сценарий 2: Поиск ошибок

**Проблема:** Приложение возвращает ошибку 500.

**Решение:**

1. В поиске выберите период (например, "Last 1 hour")
2. Нажмите на трассировку с 🔴 в столбце Errors
3. Посмотрите детали — стек ошибки и место возникновения
4. Определите проблемный сервис и исправьте ошибку

### Сценарий 3: Анализ зависимостей

**Проблема:** Нужно понять, как сервисы взаимодействуют.

**Решение:**

1. В боковом меню выберите **DAG** (Dependencies)
2. Увидите граф вызовов между сервисами
3. Размер узла = количество запросов
4. Толщина стрелки = интенсивность вызовов

```
 api-gateway ─────► authentication-service
       │                    │
       ▼                    ▼
 user-management ◄──────► database
```

## Фильтры и сортировка

### Поиск по тегам

Добавьте теги в коде для удобного поиска:

```java
// Пример добавления тегов в коде
span.tag("user.id", userId);
span.tag("quest.id", questId);
span.tag("operation", "createQuest");
```

### Фильтр ошибок

- Введите в строку поиска `error=true`
- Или ищите трассировки со статусом `500`

### Сортировка

- **Time** — по времени (по умолчанию)
- **Duration** — по времени выполнения
- **Spans** — по количеству spans

## Часто задаваемые вопросы

### Q: Почему нет трассировок?

**A:**

1. Проверьте, что Jaeger запущен: `docker ps | grep jaeger`
2. Проверьте сетевое подключение между сервисами
3. Подождите 1-2 минуты для сбора данных
4. Убедитесь, что в application.yml `sleuth.enabled: true`

### Q: Как изменить процент трассировки?

**A:**

В файле `application.yml`:

```yaml
# Development: 100%
management:
  tracing:
    sampling:
      probability: 1.0

# Production: 10%
management:
  tracing:
    sampling:
      probability: 0.1
```

### Q: Где хранятся данные Jaeger?

**A:**

В dev-режиме данные хранятся в памяти контейнера (`BADGER_EPHEMERAL: true`).
После перезапуска данные теряются.

Для persistent storage используется Elasticsearch (见 docker/jaeger/jaeger.yml).

## Рекомендации

### 1. Регулярный мониторинг

- Проверяйте Jaeger раз в неделю
- Обращайте внимание на медленные запросы
- Исправляйте проблемы до того, как они станут критическими

### 2. Алёрты (дополнительно)

Можно настроить алёрты при аномалиях:

- Время отклика > порога
- Количество ошибок > порога
- Необычное количество запросов

### 3. Тегирование

Добавляйте кастомные теги для удобного поиска:

```java
span.tag("user.id", request.getUserId());
span.tag("quest.difficulty", quest.getDifficulty().name());
span.tag("team.id", teamId);
```
